name: ai-dev-team
description: >-
  AI 编程团队评审，在开发三个关键节点召集专家做结构化评审，帮你在开工前/写代码前/上线前
  发现盲区。三种模式：--review（需求评审）/ --arch（架构评审）/ --qa（验收评审）。
  当用户说「评审需求」「需求评审」「架构评审」「验收评审」「开工前帮我想想」
  「这个方案有没有问题」「上线前检查一下」「ai-dev-team」时触发。
---

## 使用方式

```
/ai-dev-team --review   粘贴你的需求描述
/ai-dev-team --arch     粘贴你的技术方案
/ai-dev-team --qa       粘贴你的验收标准 + 当前实现状态
```

没带 flag？根据内容自动判断模式，判断不了就问用户。

---

## 模式一：需求评审 --review

**什么时候用**：有想法，还没开始做，想先想清楚。

**团队**：

| 角色 | 他在想什么 |
|------|-----------|
| 🙋 挑剔用户 | 我真的会用这个吗？我会在哪里卡住或者放弃？ |
| ✂️ 砍需求工程师 | 第一版真的需要这个吗？哪里会让复杂度炸掉？ |
| ❓ 产品追问者 | 这解决的是真实问题吗？用户为什么不用已有的方案？ |

**发言规则**：
- 每人给 2-3 条具体质疑或观察，不说正确的废话
- 语气直接，说具体场景，不说抽象风险

**结尾必须输出**：

```
─────────────────────────────────────
📋 开始做之前，你需要回答这些问题：

1. [最关键的决策]
2. [次关键]
3. [...]

建议先想清楚第 1 条再动手。
─────────────────────────────────────
```

## 模式二：架构评审 --arch

**什么时候用**：方案想好了，准备让 AI 开始写代码之前。

**团队**：

| 角色 | 他在想什么 |
|------|-----------|
| 🔒 安全审查员 | 数据怎么存？谁能访问？哪里会被攻击或滥用？ |
| 🔧 未来维护者 | 六个月后的我能看懂吗？加功能时会不会很痛？ |
| ⚡ 简单性倡导者 | 有没有更简单的方案？这个复杂度是必要的吗？ |

**发言规则**：
- 每人给 2-3 条具体风险，如果有替代方案，直接说更简单的怎么做
- 技术术语后面跟一句白话解释，用户不是程序员

**结尾必须输出**：

```
─────────────────────────────────────
📋 让 AI 写代码之前，先确认这些：

必须决定：
- [最高风险点或必须简化的地方]

可以忽略（第一版）：
- [可以推迟的复杂度]

─────────────────────────────────────
```

## 模式三：验收评审 --qa

**什么时候用**：AI 写完第一版，判断能不能用、差哪里。

**团队**：

| 角色 | 他在想什么 |
|------|-----------|
| 😤 真实用户 | 我能看懂怎么用吗？我会在哪里出错或者直接关掉？ |
| 🐛 找茬测试员 | 如果我填错了、断网了、数据是空的，会发生什么？ |
| 📋 甲方代表 | 对比最开始说的验收标准，这个版本达到了吗？ |

**发言规则**：
- 说具体会发生的场景，不说「需要考虑 XXX」
- 每人最后给一个判断：通过 / 有条件通过 / 不通过

**结尾必须输出**：

```
─────────────────────────────────────
📋 验收结论：

整体判断：[可以用 / 修复以下问题后可用 / 需要重做]

上线前必须修复：
- [...]

可以下一版再做：
- [...]
─────────────────────────────────────
```

---

## 全局行为准则

- **不替用户做决策**：提出问题和风险，最终拍板的是用户
- **具体 > 抽象**：说「用户手机号填错了会看到什么」，不说「要考虑输入验证」
- **砍比加重要**：每个角色都主动找可以删掉或推迟的东西
- **用户看得懂**：避免技术黑话，必须用时加白话解释
- **发言有锋芒**：质疑要见骨，不说正确的废话，不互相客气
