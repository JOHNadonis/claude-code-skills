# 对话沉淀与分享系统 (Distill)

> 把一次对话的所有价值榨干——经验留给自己，故事分享给别人。

---

## 与 reflect 的区别

| | reflect | distill |
|---|---------|---------|
| 侧重 | 认知层面的深度反思 | 全面结构化的经验提炼 |
| 输出 | 叙事性的认知更新 | 六维结构化总结 + 可选文章 |
| 目的 | "我的思维发生了什么变化" | "这次交互产生了什么可复用的价值" |
| 适用 | 有认知转折的对话 | 任何有信息量的对话 |

两者可以共存：先 `distill` 沉淀事实，再 `/reflect` 深挖认知。

---

## 触发方式

### 自然语言触发

- "总结一下"、"沉淀一下"、"帮我总结下这次交流"
- "梳理一下咱们聊的"、"提炼一下经验"
- "distill"、"summarize our session"

### 命令触发

| 命令 | 说明 |
|------|------|
| `/distill` | 标准沉淀（六维总结 + 持久化） |
| `/distill --article` | 沉淀 + 直接生成文章（跳过询问） |

### 推荐触发（AI 主动建议）

当检测到以下信号时，AI 在回复末尾轻量提示一句，**不自动执行**：

**触发信号**（满足任意一条即可建议）：

| 信号 | 说明 |
|------|------|
| 对话轮次 > 20 | 聊了很多，大概率有值得沉淀的内容 |
| 创建了新文件/Skill/项目 | 有具体产出物，值得记录过程 |
| 经历了方向调整 | 走过弯路，有坑可记 |
| 解决了复杂 debug | 排错过程本身就是经验 |
| 用户说了完成类感叹 | "搞定了""终于好了""折腾了半天""完成了" |
| 讨论了架构/设计决策 | 决策背后的权衡值得留档 |

**提示格式**（一行，不打断用户）：

```
💡 这次对话挺有料的，要 /distill 沉淀一下吗？
```

**约束**：
- 每次对话最多建议 1 次，不反复提醒
- 用户忽略建议后不再追问
- 纯闲聊、简单问答不触发建议

## 核心流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ ① 回溯   │───→│ ② 提炼   │───→│ ③ 持久化  │───→│ ④ 文章？  │
│ 扫描对话  │    │ 六维结构  │    │ 双层存储  │    │ 可选生成  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

## 第一步：回溯扫描

默读整个对话，标记以下内容：

- **做了什么事**：完成的任务、解决的问题、创建的产出物
- **经历了什么过程**：走过的弯路、尝试过的方案、方向调整
- **出现了什么意外**：预期与实际的差异、意外发现
- **做了什么决策**：在什么岔路口选了哪条路、为什么
- **产生了什么工具/代码/配置**：可直接复用的具体产出
- **涌现了什么认知**：新理解、新视角、观念变化

---

## 第二步：六维提炼

将扫描结果组织为以下六个维度。**不是每个维度都必须有内容**——没有就跳过，不硬凑。

### 维度 1：做了什么（What）

一句话概括 + 关键步骤列表。

```markdown
## 做了什么
> 一句话：[用一句话概括这次对话完成了什么]

- [步骤/成果 1]
- [步骤/成果 2]
- ...
```

### 维度 2：学到什么（Learn）

新知识、新技能、新理解。每条要具体，不要"学到了 XX 很重要"这种废话。

```markdown
## 学到什么
- **[知识点标题]**：[具体内容，要能让未来的自己直接用]
- ...
```

### 维度 3：踩了什么坑（Pitfall）

遇到的问题、错误的假设、浪费的时间。格式：坑 → 原因 → 正确做法。

```markdown
## 踩了什么坑
- ❌ [坑的描述]
  → 💡 原因：[为什么会踩]
  → ✅ 正确做法：[应该怎么做]
```

### 维度 4：关键决策（Decision）

在什么节点做了什么选择，以及背后的权衡。

```markdown
## 关键决策
- **[决策点]**：选了 [A]，放弃了 [B]
  → 原因：[具体权衡]
  → 回头看：[这个决策对不对？有没有更好的选择？]
```

### 维度 5：可复用模式（Pattern）

从这次经历中提炼的、下次可以直接用的模式或规则。

```markdown
## 可复用模式
- **[模式名]**：当 [触发条件] 时，[具体做法]，因为 [原因]
```

### 维度 6：未解问题（Open）

还没搞清楚的事、值得后续探索的方向。

```markdown
## 未解问题
- [ ] [问题描述] #open-question
```

## 第三步：双层持久化

### 层 1：详细文件（完整版）

**路径**：`~/.Codex/digests/YYYY-MM-DD-HH-mm-主题关键词.md`

将六维提炼的完整内容写入独立文件，模板：

```markdown
# [对话主题] 沉淀记录

> 日期：YYYY-MM-DD HH:mm
> 时长：约 [估算] 分钟
> 标签：#distill #[主题标签1] #[主题标签2]

---

## 做了什么
[内容]

## 学到什么
[内容]

## 踩了什么坑
[内容]

## 关键决策
[内容]

## 可复用模式
[内容]

## 未解问题
[内容]

---
_由 distill skill 自动生成_
```

### 层 2：MEMORY.md 摘要（精华版）

从六维内容中挑选**最值得跨会话记住的 1-3 条**，追加到 `~/.Codex/MEMORY.md`。

**写入规则**：

1. 先用 Read 读取当前 MEMORY.md
2. 判断是否有已存在的相似条目：
   - 有且一致 → 强化（追加验证案例，引用计数 +1）
   - 有但矛盾 → 标注对比，保留两个版本
   - 没有 → 新增，放到最匹配的 section 下
3. 使用 Edit 工具追加，不覆盖现有内容
4. 条目格式遵循 MEMORY.md 已有格式

**筛选标准**（什么值得写入 MEMORY.md）：

- ✅ 下次遇到类似场景能直接用的经验
- ✅ 违反直觉的发现
- ✅ 反复出现的模式
- ❌ 只在特定项目/场景有效的细节
- ❌ 显而易见的常识

### 输出格式

完成持久化后，向用户展示：

```
📦 沉淀完成

📄 详细记录：~/.Codex/digests/[文件名]
🧠 写入 MEMORY.md：
  - [条目1摘要]
  - [条目2摘要]（如果有）

要生成一篇公众号分享文章吗？
```

## 第四步：公众号文章（可选）

沉淀完成后询问用户是否生成文章。用户确认后进入文章生成流程。

### 风格自动判断

根据对话内容自动选择最合适的文章风格：

| 风格 | 适用场景 | 特征 |
|------|---------|------|
| **技术教程风** | 解决了一个技术问题、搭建了某个系统 | 步骤清晰，读者能跟着复现 |
| **故事叙述风** | 经历了曲折的排错、方向调整 | 有起承转合，带入感强 |
| **经验干货风** | 积累了通用的方法论、模式 | 结论先行，直给有用信息 |
| **对话实录风** | 人机协作过程本身有示范价值 | 保留交互过程，展示 AI 使用技巧 |

判断逻辑：
- 对话中有明确的"问题→解决"弧线 → 技术教程风 或 故事叙述风
- 对话中提炼了多条可复用经验 → 经验干货风
- 对话过程展示了有趣的人机协作模式 → 对话实录风
- 如果不确定 → 询问用户偏好

### 文章生成规则

1. **标题**：抓眼球但不标题党，体现核心价值
2. **开头**：用一个具体场景或问题切入，不要"在当今这个时代..."
3. **正文**：基于六维沉淀内容重新组织，但用人话写，不是复制粘贴结构化数据
4. **结尾**：给读者一个可带走的收获，不说"希望对你有所帮助"
5. **长度**：2000-4000 字，根据内容量弹性调整
6. **语言**：中文，口语化但不随意，有信息密度

### 文章禁止事项

- ❌ "在当今这个...的时代"
- ❌ "让我们深入探讨"
- ❌ "综上所述"、"总而言之"
- ❌ "希望这篇文章对你有所帮助"
- ❌ 赋能、抓手、打法、颗粒度等互联网黑话
- ❌ 连续三个以上结构相同的排比句
- ❌ 虚构不存在的对话细节

### 文章输出

```markdown
---
title: [文章标题]
style: [技术教程/故事叙述/经验干货/对话实录]
source: distill session [YYYY-MM-DD]
---

[文章正文]
```

文章直接输出到对话中，同时保存一份到 `~/.Codex/digests/[同名]-article.md`。

## 与现有 Skill 的关系

### 与 reflect 的协同

`distill` 做完后可以接 `/reflect`：
- `distill` 提供了"发生了什么"的事实基础
- `reflect` 在此基础上做更深层的认知反思

### 与 conversation-notes 的协同

如果当前会话有 conversation-notes 在运行：
- `distill` 可以读取 notes 文件作为回溯扫描的辅助输入
- 沉淀完成后在 notes 文件末尾追加一行链接：`→ 详细沉淀：[digests 文件路径]`

### 与写作类 Skill 的协同

如果用户对自动生成的文章不满意，可以用其他写作 Skill 重写：
- `dankoe-writer`：更有态度的深度长文
- `tk-writer`：冷幽默理性风
- `naval-writer`：格言体简洁风

## 完整示例

### 示例：一次 Skill 开发对话的沉淀

```
用户：总结一下

Codex：

📦 沉淀完成

## 做了什么
> 一句话：从需求梳理到完成，创建了 distill Skill——一个对话沉淀与分享系统。

- 梳理了 6 个设计决策（输出物、内容结构、存储位置、文章风格、触发方式、文章触发）
- 参考了 reflect / conversation-notes / dankoe-writer 三个现有 Skill 的设计
- 创建了 ~/.Codex/skills/distill/SKILL.md
- 创建了 ~/.Codex/digests/ 存储目录

## 学到什么
- **Skill 之间的定位差异化很重要**：reflect 和 distill 看似相似，
  但一个是认知反思（思维变化），一个是经验提炼（可复用的事实）。
  功能重叠的 Skill 会让触发变得模糊。

## 踩了什么坑
- ❌ 初始设计时没考虑和 reflect 的区别
  → 💡 原因：功能描述太接近
  → ✅ 正确做法：先列对比表，明确各自的不可替代性

## 关键决策
- **命名选了 distill 而非 reflect-and-share**：
  选了 distill，因为"蒸馏"更准确地描述了"从大量对话中提炼精华"的过程，
  且与已有的 reflect 命名空间不冲突。

## 可复用模式
- **需求澄清的渐进式提问**：当需求模糊时，
  一次问一个关键决策点（而非一次性列出所有问题），
  每个问题给选项，降低用户认知负担。

## 未解问题
- [ ] distill 和 reflect 的触发词是否有重叠需要处理？ #open-question

---
📄 详细记录：~/.Codex/digests/2025-02-04-14-30-distill-skill-creation.md
🧠 写入 MEMORY.md：
  - "Skill 设计要先画定位对比表再动手"

要生成一篇公众号分享文章吗？
```

用户说"写一篇"：

```
---
title: 我让 AI 给自己造了个"沉淀"按钮
style: 对话实录风
---

每次和 Codex 聊完一个项目，我都会手打一句"总结一下咱俩聊了啥"。

干了大半年，我突然觉得——这事儿出现频率这么高，为什么不做成一个自动化的 Skill？

于是就有了今天这个故事...

[文章继续]
```
