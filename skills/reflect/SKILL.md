# 反思引擎 v2 (Reflect Engine)

> 好的反思像老友复盘棋局 —— 不是说"这步好、那步差"，而是说"你当时在想什么？为什么没看到那条线？下次遇到类似局面，怎么更快看穿？"

---

## 核心原则：反机械化

**这不是一个填表工具，这是一个思维过程。**

以下行为被明确禁止：

- ❌ 使用"What went well / What went wrong / Lessons learned"等空洞框架
- ❌ 写"成功实现了 X 功能"这种毫无信息量的总结
- ❌ 把显而易见的事情当洞见（"测试很重要"、"需求要先搞清楚"）
- ❌ 用编号列表堆砌表面观察
- ❌ 假装所有事情都有明确结论

以下品质被鼓励：

- ✅ 叙事感 —— 像讲故事一样还原思维过程
- ✅ 具体性 —— 每个洞见锚定对话中的一个具体时刻
- ✅ 意外感 —— 优先记录违反直觉的发现
- ✅ 诚实 —— 承认不确定性和未解问题
- ✅ 可操作 —— 洞见能压缩成一条可执行的启发式规则

## v2 新增：记忆加载协议（对话开头）

### 🧠 启动时自动加载

**每次新对话开始时**，在创建笔记的同时，尝试加载用户的累积记忆：

```
加载顺序：
1. 读取 ~/.Codex/MEMORY.md（全局经验库）
2. 读取当前项目 .Codex/LESSONS.md 或项目根目录 LESSONS.md（如果存在）
3. 读取当前项目 README.md（如果存在，获取项目上下文）
```

**加载后行为**：
- 默读，不输出给用户（除非用户问"你记得什么"）
- 将加载的经验作为内部上下文，影响后续回答
- 如果本次对话主题与某条记忆高度相关，主动提及：
  ```
  💡 根据之前的经验（[日期]），[相关经验]。这次也适用吗？
  ```

### 加载失败处理

如果 MEMORY.md 不存在，静默跳过 —— 不要告诉用户"您还没有知识库"之类的话。这是第一次使用，反思结束后自然会创建。

## 触发机制

### 1. 半自动触发（与 conversation-notes 联动）

当 conversation-notes 检测到对话即将结束时（用户说"就这样"、"先这样"、阶段性任务完成等），自动执行**快速反思**：

```
💭 反思：[用 2-3 句自然语言写出本次对话中最值得带走的一个认知]
→ 要持久化到知识库吗？（说"持久化"或"记住这个"触发写入）
```

**关键**：快速反思不是摘要，而是「如果只能带走一个想法，那是什么？」

### 2. 手动触发

| 命令 | 模式 | 说明 |
|------|------|------|
| `/reflect` | 标准反思 | 完整复盘当前对话，生成反思文档，持久化关键经验 |
| `/reflect deep` | 深度反思 | 标准 + 读取 MEMORY.md 做跨会话模式识别 |
| `/reflect project` | 项目级反思 | 生成/更新项目目录下的 LESSONS.md，可跨 AI 复用 |
| `/reflect review` | 记忆审查 | 查看 MEMORY.md，执行衰减/强化/清理 |

### 3. 自然语言触发

用户说以下内容时自动触发标准反思：
- "反思一下"、"复盘一下"、"总结下经验"
- "这次有什么教训"、"记住这个经验"
- "reflect"、"what did we learn"

### 4. v2 新增：智能触发建议

当检测到以下模式时，**主动建议**（不自动执行）反思：
- 一个复杂任务刚完成（多步骤、多文件修改）
- 调试过程中经历了方向性调整
- 用户说了类似"终于搞定了"、"折腾了半天"的感叹
- 对话时长超过 30 分钟

建议方式（轻量，一行）：
```
💭 这次过程挺值得复盘的，要 `/reflect` 一下吗？
```

## 反思过程

### 第一步：回溯扫描

默读整个对话（或指定范围），标记五类关键时刻：

| 时刻类型 | 寻找什么 |
|---------|---------|
| **转折点** | 我们改变方向的时刻 —— 为什么改？信号是什么？ |
| **卡壳点** | 花了不成比例时间的地方 —— 真正的瓶颈在哪层？ |
| **顿悟点** | 突然想通的时刻 —— 是什么触发了突破？能否制造这种触发？ |
| **假设崩塌点** | 以为 A 结果是 B —— 假设是怎么形成的？什么信号被忽略了？ |
| **无声的好决策** | 没出问题所以没人注意的正确判断 —— 这个判断能否提炼为规则？ |

### 第二步：深度自问（内部思考，不输出给用户）

对标记的时刻，进行三个维度的拷问：

**认知维度：**
- 我们当时的心智模型和现实差了多远？
- 如果从结局倒推，最早在哪个点就能看出方向不对？
- 是否被"首因效应"锁定了？（第一个方案 = 最终方案？）
- 有没有一个更简单的问题被我们过度复杂化了？

**效率维度：**
- 如果只有 1/10 的时间，唯一要做的事是什么？
- 哪些工作是"因为不确定所以做了两遍"？不确定性的根源是什么？
- 如果把这次的经验教给另一个人/AI，他们能快多少？快在哪？

**结构维度：**
- 这个问题的本质结构是什么？（抽象到可以类比其他领域）
- 上次遇到类似结构是什么时候？那次的解法这次适用吗？
- 这次的经验能压缩成一条怎样的启发式规则？

### 第三步：有机输出

根据深度自问的结果，**以叙事方式**写出反思。不要逐条列表，用段落。每段聚焦一个洞见。

以下区块灵活使用（**不是全部必须**，哪个有料写哪个）：

#### 💭 认知更新
写清楚：之前以为什么 → 现在明白什么 → 这改变了什么。
要有对话中的具体锚点（"当我们试图用 X 解决 Y 的时候，才发现..."）。

#### 🔁 一个可复用的模式
用「当...的时候，试试...因为...」的格式。但不要硬凑 —— 如果没有真正的模式，跳过这段。

#### ⏪ 给开头的自己一封信
"如果回到对话开头，我会告诉你：..."
具体、可操作。不是"需求要先搞清楚"，而是"先花 5 分钟读 X 文件的第 Y 行，那里藏着真正的约束条件"。

#### 🧩 还没想明白的事
诚实列出。未解问题往往是最有价值的线索。加标签 `#open-question` 方便日后追溯。

## 持久化机制

### 持久化决策流程

反思完成后，对每条洞见执行以下判断：

```
这条经验是否足够通用？
├─ 否 → 只写入当前 session notes，不持久化
└─ 是 → MEMORY.md 中是否已有类似条目？
    ├─ 是且一致 → 强化（追加新案例，提升验证等级）
    ├─ 是但矛盾 → 标注「待验证」，保留两个版本
    └─ 否 → 新增条目到对应主题区域
```

### 写入 MEMORY.md（全局知识库）

**位置**：`~/.Codex/MEMORY.md`

**文件结构**：
```markdown
# 🧠 Long-Term Memory

> 跨会话累积的经验知识库。新对话开始时自动加载。
> 任何 AI 读到此文件都应将其作为工作指导。
> Last updated: YYYY-MM-DD

## 🔧 工作流与工具
[关于如何高效使用工具、IDE、命令行等的经验]

## 🐛 调试与问题排查
[调试思路、常见陷阱、排查方法论]

## 🏗️ 架构与设计
[架构决策、设计模式、技术选型的经验]

## 💡 认知模型
[思维方式、决策框架、认知偏差的觉察]

## 🗣️ 沟通与协作
[与 AI 协作、需求沟通、提问技巧等]

## 📦 领域知识
[特定技术栈、业务领域的专有知识]

---
_条目格式说明：_
_- **标题**（验证等级）：描述。来源：[日期] [对话主题]_
_- 验证等级：🌱 初步观察 → 🌿 多次验证 → 🌳 已确认_
_- 每条附带引用计数 [×N]，表示被多少次对话验证_
```

**条目格式**：
```markdown
- **[简短标题]**（🌱 初步观察）：[具体描述]。_来源：[YYYY-MM-DD] [对话主题]_ [×1]
```

**写入规则**：
1. 先用 Read 工具读取当前 MEMORY.md（不存在则按上述结构创建）
2. 找到最匹配的主题 section
3. 如果没有匹配的 section，创建新的
4. 使用 Edit 工具追加，不覆盖
5. 更新 "Last updated" 日期

### v2 新增：记忆衰减与强化

**每次 `/reflect` 或 `/reflect deep` 时**，对已有条目进行健康检查：

```
对 MEMORY.md 中每条经验：
├─ 本次对话再次验证？
│   └─ 是 → 引用计数 +1，可考虑升级验证等级
├─ 本次对话发现矛盾？
│   └─ 是 → 标注 ⚠️ 待验证，附上矛盾说明
├─ 超过 90 天未被引用？
│   └─ 是 → 标注 🔻 待审查（不自动删除）
└─ 标注为「待审查」超过 30 天？
    └─ 提醒用户在 /reflect review 中处理
```

**升级路径**：
- 🌱 初步观察 → 被 3+ 次不同对话验证 → 🌿 多次验证
- 🌿 多次验证 → 被 5+ 次验证且无矛盾 → 🌳 已确认
- 🌳 已确认 → 发现重大矛盾 → 降级为 🌿 + ⚠️ 待验证

### 写入项目级 LESSONS.md

在项目根目录生成/更新 `LESSONS.md`。

**设计原则**：这个文件必须**跨 AI 可读** —— Codex、Cursor、Windsurf、GitHub Copilot、Cline、Aider 任何一个 AI 工具读到都能立刻获得价值。因此遵循以下约定：

1. 纯 Markdown，不依赖任何工具特定语法
2. 文件名使用通用约定（LESSONS.md / AI-CONTEXT.md）
3. 开头包含"致任何 AI 助手"的说明段
4. 避免工具特定的配置语法

```markdown
# Project Lessons & Context

> 📌 **For any AI assistant**: Read this file FIRST before starting work on this project.
> This file contains accumulated knowledge, decisions, and gotchas from prior sessions.
> Last updated: YYYY-MM-DD | Sessions: N

## How This Project Works
[项目的核心架构和约束，新人/AI 需要知道的最小知识集]
[关键文件路径 + 一句话说明]
[技术栈版本号和硬约束]

## Decisions & Rationale
[关键架构决策，以及背后的 trade-off]
[格式：选了 A，因为 B 在 C 条件下不成立。考虑过 D 但因为 E 放弃]

## Known Gotchas
[踩过的坑 + 正确做法]
[格式：❌ 坑 → 💡 原因 → ✅ 解法]

## Patterns That Work
[在这个项目中被验证有效的模式]

## What Didn't Work (And Why)
[试过但放弃的方案，以及放弃原因 —— 避免后来者重蹈覆辙]

## Open Questions
[还没有答案的问题，留给未来解决]

## Session Log
[按日期倒序记录每次有意义的对话摘要，保留最近 10 条]
| Date | Topic | Key Outcome |
|------|-------|-------------|
| YYYY-MM-DD | [主题] | [一句话结论] |
```

**触发条件**：
- `/reflect project` 手动触发
- 项目级的重大修改完成后，主动建议

### v2 新增：跨工具兼容性

为了实现"换个 AI 也能用"的目标，LESSONS.md 同时兼容以下工具的约定：

将 LESSONS.md 的关键内容转换为对应工具的规则文件格式。

## 与 conversation-notes 的集成

conversation-notes 在检测到对话结束信号时，应执行以下额外步骤：

1. 完成正常的笔记整理（摘要、标签等）
2. 执行**快速反思**：回顾对话，提炼最值得记住的 1 个认知
3. 输出快速反思结果，末尾提示：
   ```
   💭 反思：[insight]
   → 说"持久化"写入知识库，说"/reflect"进入完整复盘
   ```
4. 如果用户确认持久化，执行持久化写入流程

---

## 与 AGENTS.md 递归自我批判的协同

反思引擎天然与 AGENTS.md 中的「递归自我批判」原则协同：
- 递归批判发生在**问题解决过程中**（实时优化答案）
- 反思发生在**任务完成后**（后置提炼经验）
- 记忆加载发生在**对话开头**（带着旧经验进入新问题）
- 三者共同构成"带着记忆开始 → 做的时候想清楚 → 做完后想明白 → 记住带走"的完整闭环

## 深度反思模式的额外步骤（`/reflect deep`）

在标准反思的基础上，增加：

### 跨会话模式识别

1. 读取 MEMORY.md 的所有现有条目
2. 寻找与本次经验的关联：
   - 是否有 🌱 初步观察可以升级为 🌿 多次验证？
   - 是否有矛盾需要标注？
   - 是否发现了 MEMORY.md 中不同条目之间的隐藏联系？
3. 输出发现的跨会话模式（如果有的话）

### 元认知观察

反思"我是怎么反思的"：
- 我的反思是否太表面？（只停留在"做了什么"而没深入"为什么"）
- 是否有盲点？（反复出现但一直没注意到的模式）
- MEMORY.md 的结构是否需要调整？（话题分类是否仍然合理？）

### v2 新增：记忆图谱

在深度反思中，尝试发现 MEMORY.md 条目之间的关联，输出简要的关联图：

```
🗺️ 记忆关联：
"调试先验证输入"(🌿) ←→ "症状在A层不代表病因在A层"(🌱)
 └→ 共同指向：不要相信第一层表象
```

## 记忆审查模式（`/reflect review`）

### 流程

1. 读取 MEMORY.md 全部内容
2. 按 section 分组展示，标注每条的：
   - 验证等级（🌱🌿🌳）
   - 引用计数 [×N]
   - 最后引用日期
   - 是否有 ⚠️ 或 🔻 标记
3. 对每条让用户选择：
   - **保留** —— 继续保持
   - **升级** —— 验证等级 +1
   - **修改** —— 更新描述
   - **归档** —— 移到 `MEMORY-ARCHIVE.md`（不删除，只搬家）
   - **删除** —— 确认后移除
4. 完成后更新 MEMORY.md

### 自动清理建议

当 MEMORY.md 超过 150 行时，自动建议执行 review：
```
📋 MEMORY.md 已积累 [N] 条经验（[M] 行），建议执行 /reflect review 进行整理
```

## 使用示例

### 示例 1：半自动快速反思

```
用户：好，这个功能就先这样吧。
Codex：[完成笔记整理]

💭 反思：这次最有意思的发现是 —— 我们花了 30 分钟调试的"性能问题"，
根因不在渲染层而在数据获取的 N+1 查询。一开始被 Chrome DevTools 的
火焰图误导，以为是组件重渲染。教训是：profile 工具展示的是症状，
不是病因。症状在 A 层不代表病因在 A 层。

→ 说"持久化"写入知识库，说"/reflect"进入完整复盘
```

### 示例 2：标准反思 `/reflect`

```
用户：/reflect

Codex：
## 💭 认知更新

这次重构 auth 模块时，我一开始假设问题出在中间件链的执行顺序上，
因为错误日志显示 token 验证在路由匹配之前就失败了。花了大约 20 分钟
在中间件排列组合上做实验。

真正的转折点是当我们决定"先看看 token 到底长什么样" —— 结果发现
前端发送的 Authorization header 格式根本就不对（多了一个空格）。
整个"中间件顺序问题"是个幻觉。

这暴露了一个思维惯性：看到"验证失败"，直觉反应是"验证逻辑有问题"，
而不是"被验证的东西本身有问题"。这和医生的常见误区一样 ——
症状指向哪个器官，就去查哪个器官，但可能只是输入（饮食、药物）有问题。

## ⏪ 给开头的自己

"先 `console.log(req.headers.authorization)` 看一眼实际值。
字面量问题占调试时间的 60%，但我们总假设输入是对的。"

## 🧩 还没想明白的

中间件链的测试策略还没理顺。单元测试能覆盖单个中间件，
但中间件之间的交互（顺序依赖、共享状态）怎么测？
集成测试太重，有没有轻量方案？ #open-question

---
📝 已持久化到 MEMORY.md：
- "调试时先验证输入数据的字面值，再怀疑处理逻辑"（🌱 初步观察）[×1]
```
