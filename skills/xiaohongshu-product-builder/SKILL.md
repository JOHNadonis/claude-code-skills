# 小红书产品打造养成系

这个 skill 帮助你完成从「在小红书发现需求」到「用 Codex 开发 MVP」的完整流程。

## 核心理念

```
小红书负责：需求发现 + 流量验证 + 用户聚集
微信小程序负责：产品交付 + 复购沉淀 + 长期变现
AI编程负责：快速开发 + 持续迭代
```

## 何时使用这个 Skill

- 你想找一个适合自己做的产品方向
- 你已经有了想法，想验证需求是否真实
- 你发了帖子，想分析评论区的反馈
- 你验证完需求，准备开始开发 MVP
- 你想复盘整个流程，优化下一轮迭代

## 工作流程

### 阶段一：需求挖掘
### 阶段二：需求评估与打分
### 阶段三：发帖验证
### 阶段四：数据分析
### 阶段五：MVP 开发

---

### 阶段一：需求挖掘（用户选择1）

#### 1.1 收集用户背景

```
为了帮你找到最适合的方向，我需要了解一些信息：

📌 你的身份/职业是什么？
   （例：程序员、宝妈、大学生、上班族...）

📌 你日常接触最多的人群是？
   （例：同事都是互联网人、身边宝妈很多、学生党朋友多...）

📌 你自己最近在生活中遇到什么反复出现的麻烦事？
   （例：每周写周报很痛苦、记账总坚持不下来、不知道给孩子吃什么...）

📌 你有什么技能或知识积累？
   （例：会编程、懂育儿、了解某个行业...）
```

#### 1.2 生成搜索词清单

根据用户背景，生成定制化的搜索词清单：

```markdown
## 为你定制的小红书搜索词清单

### 🔍 第一组：通用需求词（必搜）
1. 怎么快速[用户相关场景]
2. [用户身份]必备工具
3. [用户痛点]太麻烦了
4. 有没有[用户需求]的小程序
5. 求推荐[用户场景]

### 🔍 第二组：垂直场景词（根据用户身份）
[根据用户背景生成5-10个具体搜索词]

### 📝 搜索记录表
请在搜索时填写这个表格（可以复制到备忘录）：

| 搜索词 | 笔记数量 | 爆款数据(赞/藏) | 评论区发现 | 初步判断 |
|--------|----------|-----------------|------------|----------|
|        |          |                 |            |          |
```

#### 1.3 搜索操作指南

```markdown
## 具体操作步骤

### 每个搜索词的操作流程（5分钟/词）

**Step 1：搜索**
① 打开小红书 → 点顶部搜索框
② 输入搜索词 → 先看联想词（截图保存）
③ 点击搜索 → 记录「约X万条笔记」

**Step 2：看爆款数据**
① 点「筛选」→「排序」→「最热」
② 点进前3条笔记
③ 滑到底部，记录：❤️点赞 / ⭐收藏 / 💬评论

**Step 3：挖评论区（最重要！）**
找这些关键词：
- "求" → 求链接、求教程、求分享
- "有没有" → 有没有更简单的方法
- "怎么" → 怎么做到的
- "太...了" → 太需要了、太难了

**Step 4：记录到表格**
把发现填到搜索记录表

---

⏰ 建议节奏：
- 第一天搜 5 个词
- 第二天再搜 5 个词
- 搜完发给我，我帮你分析打分
```

### 阶段二：需求评估与打分（用户选择2）

#### 2.1 收集用户的需求候选

```
请把你发现的需求候选告诉我，格式：

需求1：[描述]
- 小红书笔记数量：
- 爆款数据（赞/藏）：
- 评论区发现：

需求2：[描述]
...

或者直接把你的搜索记录表发给我。
```

#### 2.2 执行打分

对每个需求，用以下维度打分（1-3分）：

```markdown
## 需求打分表

### 📊 打分维度

| 维度 | 1分 | 2分 | 3分 |
|------|-----|-----|-----|
| **频率** | 偶尔/低频（搬家、出国） | 每周1-2次（整理错题） | 每天都会遇到（背单词） |
| **付费** | 纯免费市场 | 有产品但免费/付费少 | 已有付费产品且卖得动 |
| **竞争** | 竞品多且有头部 | 有2-3个竞品，评价中等 | 没有竞品/现有评价很差 |
| **理解** | 完全陌生的人群 | 身边人有这个需求 | 你自己就有这个需求 |
| **技术** | 复杂后端/特殊资质 | 需要AI接口但简单 | 纯前端/简单API |

### 📈 总分判断
- 12-15分：⭐⭐⭐ 强烈推荐，立刻开始
- 9-11分：⭐⭐ 可以尝试，但要谨慎
- 8分以下：⭐ 不建议，建议换方向
```

#### 2.3 输出评估报告

```markdown
## 需求评估报告

### 排名结果

| 排名 | 需求 | 频率 | 付费 | 竞争 | 理解 | 技术 | 总分 | 建议 |
|------|------|------|------|------|------|------|------|------|
| 1 | xxx | 3 | 2 | 3 | 3 | 2 | 13 | ⭐⭐⭐ 推荐 |
| 2 | xxx | 2 | 2 | 2 | 2 | 3 | 11 | ⭐⭐ 备选 |

### 推荐方向分析

**首选：[需求名称]**

为什么推荐：
- [具体原因1]
- [具体原因2]
- [具体原因3]

- [风险点]

下一步：发帖验证，继续输入 `3` 我帮你写帖子
```

## 重要原则

1. **先验证再开发**：永远不要在没有验证的情况下开发
2. **最小可行**：MVP 只做一个核心功能
3. **快速迭代**：上线比完美更重要
4. **用户驱动**：功能加减看用户反馈，不是自己想象
